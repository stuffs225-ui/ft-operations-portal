## 3. Data Architecture

### SO / Project Table

The system has **no separate `sales_orders` table**. Sales Orders (SO) are stored in the `projects` table. The `so_number` column is the SO number; `project_code` is the FT-YYYY-NNNN sequential identifier auto-generated by the `generate_project_code()` BEFORE INSERT trigger (SECURITY DEFINER since migration 073).

Key `projects` columns relevant to this audit:

| Column | Default | Notes |
|--------|---------|-------|
| `project_status` | — | `draft` → `submitted_for_approval` → `approved` / `rejected` / `sent_back_for_revision` |
| `manufacturing_location` | `'not_set'` | Must be `'saudi'` or `'dubai'` before approval (R-003) |
| `medical_items` | `'not_set'` | Must be `'yes'` or `'no'` before approval (R-004) |
| `approved_at` | null | Set at approval time |
| `approved_by` | null | Set to approver's `profiles.id` |
| `revision_reason` | null | Set when sent back |
| `rejection_reason` | null | Set when rejected |

### Execution References

WO and PN references are stored in `project_execution_references` (`reference_type: 'wo' | 'pn'`). The execution gate is computed client-side in `src/lib/executionGate.ts` using `getExecutionGateStatus()`.

## 4. Quotation → SO Conversion

### Current Flow (as implemented)

```
QuotationDetail.tsx
  → handleConvertToSO()
  → navigate(`/projects/new?fromQuotationId=${id}`)
    → ProjectNew.tsx (loads quotation prefill from URL param)
    → User completes form, clicks "Submit for Approval" or "Save as Draft"
    → supabase.from('projects').insert(...)
    → supabase.rpc('link_quotation_to_project', { p_quotation_id, p_project_id })
```

**Trigger conditions:** The "Register Sales Order" button in `QuotationDetail` is only shown when:
- `quotation_status === 'returned_to_sales'` AND
- `role` is `admin`, `operations_manager`, or `sales_user` (`CAN_CONVERT`)

### `link_quotation_to_project` RPC (Migration 071)

SECURITY DEFINER function with the following enforcements:

| Check | Enforcement |
|-------|-------------|
| Caller authenticated | `auth.uid() IS NULL` → exception |
| Caller role | Must be `admin`, `operations_manager`, or `sales_user` |
| Ownership (sales_user) | `sales_user` may only link their own quotations |
| Quotation status gate | Quotation must be in `'returned_to_sales'` |
| Duplicate prevention | Already linked to same project → idempotent return |
| Cross-link prevention | Already linked to different project → exception |
| Hot project advancement | If linked hot project exists → advances to `'won'` stage |

### `convert_quotation_to_so` RPC (Migration 067)

This function exists and was the original atomic conversion implementation. It **is no longer used by the current UI flow** — `QuotationDetail.tsx` now navigates to `ProjectNew.tsx` which calls `link_quotation_to_project` instead. The function remains in the database but is not callable from any current UI path. It is not obsolete (could be called directly via API for admin tooling), but it is no longer on the critical UI path.

### Authorization at Conversion

| Actor | Can Convert |
|-------|-------------|
| `admin` | Yes — via `CAN_CONVERT` check + RPC role check |
| `operations_manager` | Yes — same |
| `sales_user` | Yes, own quotations only |
| `sales_coordinator` | No — not in `CAN_CONVERT`; migration 086 also blocks direct status write |
| `factory_user`, `store_user`, others | No |

## 5. SO Approval Flow

### Approval Surfaces

Two separate UI surfaces provide approval actions. Both restrict to the same roles:

```typescript
// ProjectDetail.tsx (line 89)
const CAN_APPROVE: UserRole[] = ['admin', 'operations_manager'];

// AdminApprovals.tsx — implicitly, route guard + component behavior
// ApproveModal / SendBackModal / RejectModal rendered only for admin/ops roles
```

#### Surface 1: `AdminApprovals.tsx` (`/admin-approvals`)

- Dedicated approval queue — shows pending, sent-back, rejected projects
- Opens modals: `ApproveModal`, `SendBackModal`, `RejectModal`
- `ApproveModal` includes a **Department Routing** section with checkboxes (procurement, factory, store, material_qc, project_qc, dubai_afs)
- Location auto-sets `dubai_afs` checkbox; medical auto-sets `material_qc` checkbox
- **Department routing checkbox state is logged in project events but NOT persisted to any routing table**

#### Surface 2: `ProjectDetail.tsx` → Approval & Routing tab

- Inline `ApprovePanel` component — Approve / Send Back / Reject actions within the project view
- No department routing checkboxes — simpler than `AdminApprovals`
- Only visible when `project_status === 'submitted_for_approval'`

### Approval Action: What Is Written

```typescript
// Both surfaces write the same fields to projects:
supabase.from('projects').update({
  project_status: 'approved',
  manufacturing_location: location,  // 'saudi' | 'dubai' (never 'not_set')
  medical_items: medical,            // 'yes' | 'no' (never 'not_set')
  approved_at: now,
  approved_by: profile.id,
})
```

Both surfaces also call:
- `recordProjectEvent(...)` — writes to `project_timeline_events`
- `recordAuditEntry(...)` — writes to `audit_log`

### Approval State Machine (UI-enforced transitions)

```
draft
  └─→ submitted_for_approval   (ProjectNew: "Submit" button)
        ├─→ approved            (admin / operations_manager)
        │     └─→ [execution gate: WO or PN required]
        ├─→ sent_back_for_revision  (admin / operations_manager — reason required)
        │     └─→ submitted_for_approval  (sales_user resubmits — not shown in inspected code)
        └─→ rejected            (admin / operations_manager — reason required)
```

Note: `active`, `completed`, `cancelled` status values exist in the `statusBadge` map but transitions to these are not wired in the inspected components. These are likely set by other modules or direct DB operations.

## 6. Department Routing

### Current Implementation

`AdminApprovals.tsx` `ApproveModal` (lines 51–66) maintains a `routes` state object:

```typescript
const [routes, setRoutes] = useState({
  procurement: true,
  factory: true,
  store: true,
  material_qc: false,
  project_qc: true,
  dubai_afs: false,
});
```

`handleApprove()` passes `routes` to `recordProjectEvent()` as metadata:
```typescript
{ manufacturing_location: location, medical_items: medical, routing: routes }
```

**The `routes` object is NOT written to the `projects` table or any other table.** It is only embedded in the `project_timeline_events.metadata` JSON column via `recordProjectEvent`. There is no `project_department_routing` table or equivalent in the migrations inspected.

### Gap G-9A-01

| Field | Detail |
|-------|--------|
| Gap ID | G-9A-01 |
| Description | Department routing selections are display-only UI state; not persisted to a queryable table |
| Impact | Routing decisions cannot be queried per-department (e.g., "which projects are routed to Material QC?"). Other modules cannot programmatically check their own routing status. |
| Current tier | NONE — no DB storage |
| Mitigation in place | `manufacturing_location` (which determines factory/dubai split) IS persisted. Medical items flag (which drives material_qc routing) IS persisted. The department routing checkboxes are supplementary advisory UI. |
| Severity | MEDIUM — advisory, not blocking |

## 7. WO/PN Execution Gate

### Client-Side Gate (`src/lib/executionGate.ts`)

Pure function `getExecutionGateStatus()` computes gate status from project + references:

```typescript
return {
  isApproved,
  isSaudi, isDubai,
  requiresWO: isApproved && isSaudi,
  requiresPN: isApproved && isDubai,
  hasActiveWO, hasActivePN,
  canStartSaudiFactory: isApproved && isSaudi && hasActiveWO,
  canStartDubaiFollowUp: isApproved && isDubai && hasActivePN,
};
```

### UI Enforcement

`WoPnGateCard` in `ProjectDetail.tsx` shows gate status in the Overview tab. `canAddRef` is granted to `admin`, `operations_manager`, `factory_user`. No other roles can add WO/PN references.

The factory module display in `ProjectDetail` checks `manufacturing_location !== 'saudi'` to show the Dubai note, but does NOT gate access to factory records based on WO existence.

### DB Enforcement

No TIER-1 constraint prevents:
- Creating `factory_records` rows for a Saudi project without an active WO
- Creating `dubai_project_followups` rows for a Dubai project without an active PN

The WO/PN gate is enforced only at TIER-2/3 (client-side gate helper + UI display).

### Gap G-9A-02

| Field | Detail |
|-------|--------|
| Gap ID | G-9A-02 |
| Description | No DB trigger prevents factory/Dubai execution records from being created when the required WO/PN reference is absent |
| Governance rules | R-005 (WO gate), R-006 (PN gate) |
| Impact | A factory_user with direct API access or a future UI bypass could create factory records for a Saudi project without a WO number. Control Tower would not detect this unless it cross-references `project_execution_references` |
| Current tier | TIER-2/3 only |
| Suggested fix | BEFORE INSERT trigger on `factory_records` checking for active WO when `manufacturing_location = 'saudi'`; similar trigger on `dubai_project_followups` for PN |
| Severity | MEDIUM — exploitable via direct API; not currently exploitable via existing UI |

## 8. DB Enforcement Status (R-003 and R-004)

### Migration 078 — `enforce_so_approval_fields` Trigger

Status: **TIER-1 ENFORCED** ✅

```sql
-- R-003: manufacturing_location must not be 'not_set' at approval
IF NEW.manufacturing_location = 'not_set' THEN
  RAISE EXCEPTION 'SO Approval gate (R-003): Cannot approve project...'
END IF;

-- R-004: medical_items must not be 'not_set' at approval
IF NEW.medical_items = 'not_set' THEN
  RAISE EXCEPTION 'SO Approval gate (R-004): Cannot approve project...'
END IF;
```

The trigger fires `BEFORE INSERT OR UPDATE ON projects FOR EACH ROW`. It only enforces when `NEW.project_status = 'approved'` AND the status is actually changing. Non-status updates on approved projects are not re-validated.

**This resolves backlog items B-010 and B-011** from the governance rules register. These were previously TIER-3 only (UI enforcement in `ApprovePanel` / `ApproveModal` defaulting `'not_set'` to `'saudi'`/`'no'`).

### Consistency Between Approval Surfaces

Both `ApprovePanel` (ProjectDetail) and `ApproveModal` (AdminApprovals) default `manufacturing_location` to `'saudi'` and `medical_items` to `'no'` when the current value is `'not_set'`. This means in normal UI usage, `'not_set'` values are never sent to the DB during approval. Migration 078 provides belt-and-suspenders protection against direct API calls.
